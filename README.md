    <article class="markdown-body entry-content p-3 p-md-6" itemprop="text"><h1><a id="user-content-manifests" class="anchor" aria-hidden="true" href="#manifests"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Manifests</h1>
<p>This repo is a <a href="https://github.com/kubernetes-sigs/kustomize/blob/master/docs/glossary.md#bespoke-configuration">bespoke configuration</a> of kustomize targets used by kubeflow. These targets are traversed by kubeflow's CLI <code>kfctl</code>. Each target is compatible with the kustomize CLI and can be processed indendently by kubectl or the kustomize command.</p>
<h2><a id="user-content-organization" class="anchor" aria-hidden="true" href="#organization"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Organization</h2>
<p>Various subdirectories within the repo contain a kustomize target (base or overlay subdirectory). Overlays are used for a variety of purposes such as platform resources. Both base and overlay targets are processed by kfctl during generate and apply phases and is detailed in <a href="#kfctl-processing">Kfctl Processing</a>.</p>

<h2><a id="user-content-kfctl-processing" class="anchor" aria-hidden="true" href="#kfctl-processing"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Kfctl Processing</h2>
<p>Kfctl traverses directories under manifests to find and build kustomize targets based on the configuration file <code>app.yaml</code>. The contents of app.yaml is the result of running kustomize on the base and specific overlays in the kubeflow <a href="https://github.com/kubeflow/kubeflow/tree/master/bootstrap/config">config</a> directory. The overlays reflect what options are chosen when calling <code>kfctl init...</code>.  The kustomize package manager in kfctl will then read app.yaml and apply the packages, components and componentParams to kustomize in the following way:</p>
<ul>
<li><strong>packages</strong>
<ul>
<li>are always top-level directories under the manifests repo</li>
</ul>
</li>
<li><strong>components</strong>
<ul>
<li>are also directories but may be a subdirectory in a package.</li>
<li>a component may also be a package if there is a base or overlay in the top level directory.</li>
<li>otherwise a component is a sub-directory under the package directory.</li>
<li>in all cases a component's name in app.yaml must match the directory name.</li>
<li>components are output as <code>&lt;component&gt;.yaml</code> under the kustomize subdirectory during <code>kfctl generate...</code>.</li>
<li>in order to output a component, a kustomization.yaml is created above the base or overlay directory and inherits common parameters, namespace and labels of the base or overlay. Additionally it adds the namespace and an application label.</li>
</ul>
</li>
</ul>
<pre><code>apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
bases:
  - &lt;component&gt;/{base|overlay/&lt;overlay&gt;}
commonLabels:
  app.kubernetes.io/name: &lt;appName&gt;
namespace:
  &lt;namespace&gt;
</code></pre>
<ul>
<li><strong>component parameters</strong>
<ul>
<li>are applied to a component's params.env file. There must be an entry whose key matches the component parameter. The params.env file is used to generate a ConfigMap. Entries in params.env are resolved as kustomize vars or referenced in a deployment or statefulset env section in which case no var definition is needed.</li>
</ul>
</li>
</ul>
<p>Multiple overlays -</p>
<p>Kfctl has the capability to combine more than one overlay during <code>kfctl generate ...</code>. An example is shown below where the profiles target in <a href="https://github.com/kubeflow/manifests/tree/master/profiles">manifests</a> can include either debug changes in the Deployment or Device information in the Namespace (the devices overlay is not fully integrated with the Profile-controller at this point in time and is intended as an example) or <strong>both</strong>.</p>
<pre><code>profiles
├── base
│   └── kustomization.yaml
└── overlays
    ├── debug
    │   └── kustomization.yaml
    └── devices
        └── kustomization.yaml
</code></pre>
<h4><a id="user-content-what-are-multiple-overlays" class="anchor" aria-hidden="true" href="#what-are-multiple-overlays"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>What are Multiple Overlays?</h4>
<p>Normally kustomize provides the ability to overlay a 'base' set of resources with changes that are merged into the base from resources that are located under an overlays subdirectory. For example
if the kustomize <a href="https://github.com/kubernetes-sigs/kustomize/blob/master/docs/glossary.md#target">target</a> is named foo there will be a foo/base and possibly one or more overlays such as foo/overlays/bar. A kustomization.yaml file is found in both foo/base and foo/overlays/bar. Running <code>kustomize build</code> in foo/base will generate resources as defined in kustomization.yaml. Running <code>kustomize build</code> in foo/overlays/bar will generate resources - some of which will overlay the resources in foo/base.</p>
<p>Kustomize doesn't provide for an easy way to combine more than one overlay for example foo/overlays/bar, foo/overlays/baz. However this is an open feature request in kustomize <a href="https://github.com/kubernetes-sigs/kustomize/issues/759">issues</a>. The ability to combine more than one overlay is key to handling components like tf-job-operator which has several types of overlays that can 'mix-in' whether a TFJob is submitted to a namespace or cluster-wide and whether the TFJob uses gang-scheduling.</p>
<h4><a id="user-content-merging-multiple-overlays" class="anchor" aria-hidden="true" href="#merging-multiple-overlays"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Merging multiple overlays</h4>
<p>Since each overlay includes '../../base' as its base set of resources - combining several overlays where each includes '../../base' will cause <code>kustomize build</code> to abort, complaining that it recursed on base. The approach is to create a kustomization.yaml at the target level that includes base and the contents of each overlay's kustomization file. This requires some path corrections and some awareness of the behavior of configMapGenerator, secretMapGenerator and how they are copied from each overlay. This kustomization.yaml can be constructed manually, but is integrated within kfctl via the app.yaml file. Using tf-job-operator as an example, if its componentParams has the following</p>
<pre><code>  componentParams:
    tf-job-operator:
    - name: overlay
       value: cluster
    - name: overlay
    - value: gangscheduled
</code></pre>
<p>Then the result will be to combine these overlays eg 'mixin' an overlays in the kustomization.yaml file.</p>
<h4><a id="user-content-merging-multiple-overlays-to-generate-appyaml" class="anchor" aria-hidden="true" href="#merging-multiple-overlays-to-generate-appyaml"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Merging multiple overlays to generate app.yaml</h4>
<p>Normally when <code>kfctl init ...</code> is called it will download the kubeflow repo under <code>&lt;deployment&gt;/.cache</code> and read one of the config files under <code>.cache/kubeflow/&lt;version&gt;/bootstrap/config</code>. These config files define packages, components and component parameters (among other things). Each config file is a compatible k8 resource of kind <em>KfDef</em>. The various config files are:</p>
<ul>
<li>kfctl_default.yaml</li>
<li>kfctl_basic_auth.yaml</li>
<li>kfctl_iap.yaml</li>
</ul>
<p>Both kfctl_basic_auth.yaml and kfctl_iap.yaml contain the contents of kfctl_default.yaml plus some additional changes specific to using basic_auth when the cluster is created and platform specific resources if the platform is <strong>gcp</strong>. This PR corrects this redundancy by using kustomize to combine a <strong>gcp</strong> overlay and/or a <strong>basic_auth</strong> overlay. Additionally, due to pipeline refactoring, the kustomize package manager has split the bundled pipeline component in ksonnet to a set of individual pipeline targets. This results in the following:</p>
<pre><code>config
├── base
│   └── kustomization.yaml
└── overlays
    ├── basic_auth
    │   └── kustomization.yaml
    ├── gcp
    │   └── kustomization.yaml
    ├── ksonnet
    │   └── kustomization.yaml
    └── kustomize
        └── kustomization.yaml
</code></pre>
<p>Based on the cli args to <code>kfctl init...</code>, the correct overlays will be merged to produce an app.yaml.
The original files have been left as is until UI integration can be completed in a separate PR</p>
